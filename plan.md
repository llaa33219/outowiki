# OutoWiki 라이브러리 설계 문서

## 에이전트 지식의 모든 것을 기록하는 위키 시스템

---

## 1. 개요 및 철학

### 1.1 프로젝트 정의

OutoWiki는 AI 에이전트의 기억, 학습, 역사를 체계적으로 관리하기 위해 설계된 파이썬 라이브러리이다. 이 라이브러리의 핵심 철학은 단순명료하다. 에이전트가 생성하는 모든 정보는 Wiki라는 형태의 구조화된 저장소에 축적되며, 이 저장소는 에이전트 자신의 작업 공간이자 기억의 보고로서 기능한다. 기존 AI 메모리 시스템이 단순히 정보를 저장하고 검색하는 데 그쳤다면, OutoWiki는 실제 위키의 운영 원칙을 완전히 수용하여 문서의 분류, 병합, 분할, 참조, 수정이라는 일련의 과정을 자동화한다.

이 접근법의 차별점은 명확하다. Andrej Karpathy가 제시한 LLM Wiki 개념과 Hermes Agent의 자체 학습 메커니즘에서 영감을 받았으나, OutoWiki는 더 범용적이고 구조적인 설계로 나아간다. Hermes Agent가 스킬 생성에 집중한다면, OutoWiki는 에이전트의 모든 종류의 지식—대화 기록, 도구 사용법, 사용자 모델, 학습 결과—을 하나의 통합된 위키 체계 아래 조직한다.

### 1.2 핵심 설계 원칙

OutoWiki는 다섯 가지 핵심 원칙 위에 구축된다. 첫째, 직관적 분류 체계 원칙이다. 모든 문서는 폴더 기반의 계층적 분류를 따르며, 폴더 이름 자체가 카테고리 의미론을 담는다. 예를 들어 `사용자/취향/음식` 폴더는 사용자 취향 중 음식 관련 정보를 의미하며, 하위 폴더를 통해 더 구체적인 하위 분류가 가능하다. 이 구조는 AI가 토큰을 낭비하지 않고 필요한 문서를 빠르게 찾을 수 있게 한다.

둘째, 자체 문서화 원칙이다. OutoWiki의 Wiki는 단순한 저장소가 아니라 자체 진화하는 문서 체계다. 새로운 정보가 들어올 때마다 시스템은 기존 문서와의 관계를 분석하고, 필요시 문서를 생성하거나 수정하거나 병합한다. 이 과정은 전적으로 내부 에이전트에 의해 자율적으로 수행된다.

셋째, 가시적 기억 원칙이다. Co-Wiki 아키텍처에서 영감을 받아, OutoWiki는 인간과 AI 모두에게 읽기 쉬운 마크다운 형식을 채택한다. 벡터 데이터베이스 기반의 차가운 저장소와 달리, 위키의 각 페이지는 직접 열람 가능하며 문서 간의 관계는 백링크와 카테고리를 통해 명시적으로 표현된다.

넷째, 토큰 효율성 원칙이다. MemGPT의 계층적 메모리 관리와 유사하게, OutoWiki는 컨텍스트 윈도우의 제약을 인식하고 문서를 효과적으로 분할하여 필요한 정보만 로드한다. RAG 시스템과 달리, OutoWiki는 임베딩 기반 검색의 지연과 오버헤드를 피하며 대신 파일 시스템 네비게이션을 통해 직접적으로 문서에 접근한다.

다섯째, 폐쇄적 학습 루프 원칙이다. Hermes Agent의 학습 메커니즘에서 영감을 받아, OutoWiki는 에이전트가 경험에서 학습하고 지식을 개선하는 완전한 루프를 구현한다. 정보 기록, 분석, 계획, 반영이라는 단계를 통해 시스템은 지속적으로 자체 개선한다.

### 1.3 기존 시스템과의 비교

OutoWiki의 포지셔닝을 명확히 하기 위해 기존 접근법과의 차이를 비교해보자. 전통적인 RAG 시스템은 문서를 청크로 분할하고 임베딩을 생성한 후 벡터 데이터베이스에 저장한다. 이 방식은 대규모 지식 베이스에 적합하지만, 청크 경계의 인공성, 검색 실패 가능성, 인프라 복잡성이라는 문제를 안고 있다. 반면 OutoWiki는Wiki 문서를 유기적으로 분할하며, 이는 인간의 인지 구조를 반영한 자연스러운 청킹이다.

MemGPT의 계층적 메모리 시스템은 RAM과 디스크의 비유를 사용하여 주 메모리와 외부 메모리를 구분한다. OutoWiki도 유사한 계층 개념을 채택하되, Wiki 구조를 통해 구현한다는 점에서 차이가 있다. Hermes Agent의 스킬 시스템은 에이전트의 절차적 기억을 파일로 저장하지만, OutoWiki는 스킬뿐 아니라 에피소드 기억, 의미 기억, 사용자 모델 등 모든 유형의 기억을 통합 위키로 관리한다.

Mem0은 가장 널리 사용되는 AI 에이전트 메모리 플랫폼으로, 벡터 저장소와 그래프 데이터베이스를 결합한 하이브리드 메모리 아키텍처를 제공한다. 의미 기억, 에피소드 기억, 절차 기억을 구분하여 관리하며, ADD/UPDATE/DELETE/NOOP 연산을 통해 자동으로 정보를 업데이트한다. 그러나 저장 방식이 벡터/그래프 기반이므로 인간이 직접 읽기 어렵고, 문서 구조가 아닌 개별 기억 단위로 관리되어 대규모 문서 관리에는 한계가 있다.

MemFree는 하이브리드 AI 검색 엔진으로, 텍스트/이미지/파일/웹페이지 검색과 다중 AI 모델(GPT, Claude, Gemini) 지원을 제공한다. 지식 베이스와 인터넷 검색을 결합한 하이브리드 검색 방식을 사용하지만, 에이전트의 지속적인 학습이나 자체 문서화 기능은 지원하지 않는다.

MemU는 NevaMind-AI에서 개발한 24/7 사전적 에이전트를 위한 오픈소스 메모리 프레임워크이다. 파일 시스템比喻을 사용하여 Resource Layer, Item Layer, Category Layer의 3레이어 계층 구조를 구현한다. 이중 모드 검색(RAG 기반 빠른 검색 + LLM 기반 심층 추론)을 지원하며, Locomo 벤치마크에서 92% 정확도와 기존 대비 90% 토큰 비용 절감을 달성했다. 그러나 PostgreSQL/pgvector 기반 저장소로 인해 인간이 직접 읽기 어렵고, 문서 구조가 아닌 개별 기억 단위로 관리된다. 또한 자체 학습이나 Wiki 규칙은 지원하지 않는다.

### 1.4 라이브러리 탑제 가정

OutoWiki는 독립 실행이 아닌 호스트 에이전트에 탑제되는 구성 요소로 설계된다. 이 가정은 라이브러리의 활용 방식을 결정한다. 첫째, LLM 도구로의 탑제가 가능하다. `search()`, `record()` 등 메서드가 그대로 도구로 노출되어 에이전트가 직접 호출할 수 있다. 둘째, 프롬프트 첨부가 가능하다. 시스템 지침에 라이브러리 사용법을 포함시켜 에이전트가 자연스럽게 활용할 수 있다. 셋째, 상태 공유가 용이하다. 호스트 에이전트의生命周期과 동기화되어 일관된 메모리 관리가 가능하다.

### 1.5 제약 없는 입력 처리

OutoWiki의 입력 시스템은 형식 제약이 없다. 다양한 형태의 문자열—JSON, 일반 텍스트, 마크다운, 대화 로그 등—을 그대로受理하고 내부 에이전트가 이를 분석하여 적절히 처리한다. LLM 기반 내부 처리 덕분에 입력 형식에 구애받지 않고 유연하게 대응할 수 있다.

## 2. 시스템 아키텍처

### 2.1 전체 구조 개요

OutoWiki는 세 가지 상호작용 모듈과 하나의 Wiki 저장소로 구성된다. 설정 모듈은 LLM 공급자 연결과 Wiki 디렉토리 위치를 초기화한다. 기록 모듈은 다양한 형태의 입력 정보를 받아 내부 에이전트를 통해 Wiki 문서로 변환한다. 검색 모듈은 쿼리를 분석하여 관련 문서를 찾아 절대 경로를 반환한다. Wiki 저장소는 이 모든 문서를 마크다운 파일과 폴더 구조로 물리적으로 저장한다.

```
┌─────────────────────────────────────────────────────────────┐
│                      OutoWiki 라이브러리                      │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│  │   설정 모듈  │  │   기록 모듈  │  │   검색 모듈  │        │
│  │  (Config)   │  │  (Recorder) │  │  (Searcher) │        │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘        │
│         │                │                │                │
│         └────────────────┼────────────────┘                │
│                          │                                 │
│                   ┌──────▼──────┐                          │
│                   │  내부 에이전트  │                         │
│                   │  (Sub-Agent) │                         │
│                   └──────┬──────┘                          │
│                          │                                 │
│                   ┌──────▼──────┐                          │
│                   │  Wiki 저장소  │                         │
│                   │ (디렉토리/MD) │                         │
│                   └─────────────┘                          │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 설정 시스템 설계

OutoWiki의 설정 시스템은 두 가지 주요 영역으로 나뉜다. 첫째는 LLM 공급자 설정이다. OutoWiki는 다양한 LLM 공급자를 지원하며, 현재 OpenAI 호환 형식과 Anthropic을 기본으로 지원한다. 각 공급자에 대해 base URL, API 키, 사용할 모델, 출력 토큰 제한을 설정할 수 있다. 이러한 추상화된 설정은 사용자가 공급자를 쉽게 전환할 수 있게 하며, 테스트 환경과 프로덕션 환경의 분리도 용이하다.

둘째는 Wiki 디렉토리 설정이다. Wiki가 저장될 루트 디렉토리를 지정하며, 이 디렉토리 아래에 모든 분류 폴더와 문서가 생성된다. 설정 시 초기 분류 구조를 템플릿으로 제공할 수 있어, 사용자는 빈 Wiki에서 시작하거나 사전 정의된 분류 체계에서 시작할 수 있다.

설정 파일의 구조는 다음과 같이 YAML 또는 JSON 형식을 지원한다. providers 섹션에서는 각 LLM 공급자의 연결 정보를 정의하고, wiki 섹션에서는 Wiki 루트 디렉토리와 선택적 초기화 템플릿을 지정한다. settings 섹션에서는 문서 분할 기준, 백링크 생성 규칙, 문서 병합 임계값 등 Wiki 운영 정책을 정의한다.

### 2.3 Wiki 저장소 구조

OutoWiki의 Wiki 저장소는 폴더 기반 계층 구조를 따른다. 각 폴더는 하나의 분류 카테고리를 의미하며, 폴더 이름은 해당 카테고리의 의미론적 의미를 담는다. 폴더 내에는 마크다운 문서와 하위 분류 폴더가 공존할 수 있다. 이 구조는 Wikipedia의 카테고리 시스템과 유사하지만, 폴더라는 물리적 구조를 사용함으로써 파일 시스템 네이티브의 탐색을 가능하게 한다.

루트 디렉토리 아래에 위치하는 주요 분류는 에이전트의 지식 영역을 반영한다. 사용자 관련 정보는 `users/` 폴더 아래 조직되며, 각 사용자에 대한 하위 폴더가 생성된다. 에이전트의 도구 및 스킬에 대한 지식은 `tools/` 폴더에 저장된다. 에이전트 자체에 대한 인식과 학습 결과는 `agent/` 폴더에 기록된다. 일반적인 지식과 사실은 `knowledge/` 폴더에 분류된다. 에이전트와 사용자의 대화 기록은 `history/` 폴더에 시계열로 정리된다.


## 3. 정보 기록 시스템

### 3.1 기록 파이프라인 개요

OutoWiki의 정보 기록 시스템은 세 가지 단계로 구성된다. 첫 번째 단계는 정보 분석이다. 다양한 형태의 원시 입력—대화 로그, 에이전트 내부 사고, 외부 문서, 구조화된 데이터—을 받아 시스템이 해당 정보의 유형, 의미, 중요도를 분석한다. 두 번째 단계는 수정 계획이다. 분석 결과를 바탕으로 기존 문서를 수정할지, 새로운 문서를 생성할지, 문서를 병합할지, 분할할지를 결정하고 실행 계획을 수립한다. 세 번째 단계는 문서 반영이다. 수립된 계획에 따라 Wiki 문서를 실제로 생성, 수정, 삭제, 병합, 분할한다.

이 세 단계는 모두 내부 에이전트에 의해 수행된다. 에이전트는 설정에서 정의한 LLM 공급자를 사용하여 추론하고 판단한다. 각 단계는 독립적으로 실행 가능하며, 복잡한 입력의 경우 세 단계가 순차적으로 실행되어 최종 문서 반영까지 이어진다.

### 3.2 단계별 상세 설계

#### 3.2.1 정보 분석 단계

정보 분석 단계의 목표는 원시 입력을 구조화된 분석 결과로 변환하는 것이다. 입력은 네 가지 주요 유형으로 분류된다. 대화 기록 유형은 사용자와 에이전트 사이의 JSON 형태의 대화 로그이거나 일반 텍스트 형태의 대화 기록이다. 에이전트 내부 유형은 에이전트의 루프 내에서 생성된 중간 결과, 사고 과정, 도구 호출 결과 등을 포함한다. 외부 문서 유형은 웹 검색 결과, 파일 내용, API 응답 등 외부 소스의 정보이다. 구조화 데이터 유형은 JSON, CSV, 데이터베이스 레코드 등 구조화된 형태의 정보이다.

분석기는 각 입력에 대해 일곱 가지 속성을 추출한다. 첫째, 정보 유형은 입력이 어떤 범주에 속하는지를 나타낸다. 둘째, 핵심 주제는 정보의 가장 중요한 주제나 개념이다. 셋째, 구체적 내용은 주제를 뒷받침하는 세부 사실과 데이터이다. 넷째, 기존 지식과의 관계는 현재 Wiki에 존재하는 문서들과의 관련성이다. 다섯째, 시간적 범위는 정보의 유효 기간과 시의성이다. 여섯째, 신뢰도 점수는 정보의 출처와 정확성에 대한 평가이다. 일곱째, 중요도 점수는 Wiki에 기록할 가치가 있는지에 대한 판단이다.

분석 결과는 구조화된 딕셔너리로 표현되며, 이는 다음 단계인 수정 계획에 입력으로 전달된다.

#### 3.2.2 수정 계획 단계

수정 계획 단계는 분석 결과를 바탕으로 Wiki 문서를 어떻게 변경할지를 결정한다. 이 단계의 핵심은 기존 문서와의 중복과 충돌을 처리하는 것이다.

계획은 다섯 가지 유형 중 하나를 선택하거나 조합한다. 생성 계획은 새로운 문서를 Wiki에 추가한다. 이 경우 문서의 위치 분류 폴더, 제목, 초기 콘텐츠, 관련 문서 참조를 포함한다. 수정 계획은 기존 문서에 새 정보를 통합한다. 어떤 문서를 어떤 부분에 어떻게 추가 변경할지를 지정한다. 병합 계획은 관련 있는 여러 문서를 하나의 문서로 통합한다. 어떤 문서들을 어떤 기준으로 합칠지, 통합 후 원본 문서는 어떻게 처리할지를 결정한다. 분할 계획은 너무 커진 문서를 여러 문서로 분리한다. 분할 기준과 새 문서들의 구조를 정의한다. 삭제 계획은 더 이상 유효하지 않거나 중복된 문서를 제거한다.

계획 수립 시 에이전트는 여러 가지 고려사항을权衡한다. 문서 크기 관리 측면에서, 단일 문서의 토큰 수가 특정 임계값을 초과하면 분할을 고려한다. 일반적으로 4000 토큰을 초과하는 문서는 분할 대상이 된다. 중복 최소화 측면에서, 새로운 정보가 기존 문서와 상당 부분 중복될 경우 기존 문서를 수정하는 것이 바람직하다. 그러나 새로운 정보가 기존 정보를 대체하거나 크게 수정해야 할 때는 기존 문서를 업데이트한다. 참조 무결성 측면에서, 문서 삭제나 병합 시 해당 문서를 참조하는 다른 문서의 백링크가 깨지지 않도록 주의한다. 백링크가切れる 경우 해당 문서들을 함께 업데이트하거나 리다이렉션을 설정한다.

#### 3.2.3 문서 반영 단계

문서 반영 단계는 수립된 계획을 실제로 실행한다. 각 계획 유형에 대해 다음과 같은 구체적 작업이 수행된다.

생성 작업의 경우, 지정된 경로에 마크다운 파일을 생성하고 초기 헤더와 콘텐츠를 작성한다. 폴더가 존재하지 않으면 함께 생성한다. 새로 생성된 문서는 관련 문서의 역링크 목록에 자동으로 추가된다. 수정 작업의 경우, 기존 파일을 읽고 새로운 정보를 적절한 위치에 삽입하거나 기존 내용을 업데이트한다. 파일 수정 후 관련된 백링크 정보를 업데이트한다. 병합 작업의 경우, 병합 대상 문서들을 모두 읽고 내용을 통합한다. 새 문서를 생성하고 역링크 정보를 수집하여 새 문서에 반영한다. 원본 문서는 삭제하거나 리다이렉션 파일로 대체한다. 분할 작업의 경우, 원본 문서를 분할 기준에 따라 여러 부분으로 나누고 각각 새 문서로 저장한다. 분할된 문서들은 서로를 참조하는 크로스 링크를 포함하며, 원본 문서는 분할 사실을 기록하는 개요 문서로 변환되거나 삭제된다. 삭제 작업의 경우, 삭제 대상 문서를 읽고 역링크 목록을 확인한다. 역링크를 가진 문서들을 찾아 해당 참조를 제거하거나 업데이트한 후 원본 문서를 삭제한다.

모든 반영 작업은 원자적으로 실행되어 오류 발생 시 롤백 가능한 구조를 갖는다. 또한 모든 변경 사항은 변경 로그에 기록되어 이전 상태로의 복원이 가능하다.

### 3.3 기록 시스템 프롬프트 설계

내부 에이전트가 정보를 분석하고 계획을 수립할 때 사용하는 프롬프트는 체계적으로 설계되어야 한다. 분석 프롬프트는 입력을 받고 분석 결과를 구조화된 형식으로 출력하도록 지시한다. 계획 수립 프롬프트는 분석 결과와 현재 Wiki 상태를 입력으로 받고, 최적의 계획을 출력한다. 문서 생성 프롬프트는 계획에 따라 구체적인 마크다운 문서 콘텐츠를 생성한다.

프롬프트 설계의 핵심 원칙은 명확성과 구체성이다. 각 단계에서 기대하는 출력 형식을 명시적으로 제시하고,Wiki 문서의 구조와 규칙을 설명한다. 또한 에이전트가 결정할 사항과 사용자가 지정해야 할 사항을 구분하여 불필요한 불확실성을 줄인다.

## 4. 정보 검색 시스템

### 4.1 검색 파이프라인 개요

OutoWiki의 정보 검색 시스템도 세 가지 단계로 구성된다. 첫 번째 단계는 의도 분석이다. 검색 쿼리와 맥락 정보를 받아 시스템이 사용자가 어떤 정보를 원하는지 분석한다. 두 번째 단계는 문서 탐색이다. 분석 결과를 바탕으로 Wiki 분류 체계를 따라가며 관련 문서를 찾는다. 세 번째 단계는 문서 반환이다. 찾은 문서의 절대 경로를 반환하거나, 요청 시 문서 내용을 함께 반환한다.

검색 시스템의 핵심 설계 원칙은 토큰 효율성이다. 전체 Wiki를 컨텍스트에 로드하는 대신, 분류 체계를 통해 필요한 하위 집합만 접근한다. 이는 RAG 시스템의 임베딩 검색보다 예측 가능하고 일관된 검색 결과를 제공하며, 검색 지연 없이 즉각적인 응답이 가능하다.

### 4.2 단계별 상세 설계

#### 4.2.1 의도 분석 단계

의도 분석 단계는 검색 쿼리의 의도를 파악하고 검색 전략을 결정한다. 입력은 다양한 형태로 제공될 수 있다. 자연어 질문은 사용자가 일반 언어로 질문하는 형태이다. 구조화된 쿼리는 필터 조건과 검색 대상을 지정하는 형태이다. 대화 맥락은 현재 에이전트의 작업 상태와 관련된 정보 요청이다. 에이전트 내부 쿼리는 에이전트가 스스로의 지식이나 상태를 확인하는 요청이다.

분석 결과는 다섯 가지 속성으로 표현된다. 첫째, 정보 유형은 원하는 정보가 어떤 범주에 속하는지를 나타낸다. 사용자 프로필, 도구 사용법, 과거 대화, 사실적 지식 등이다. 둘째, 구체성 수준은 검색 범위의 폭을 결정한다. 매우 구체적인 질문은 정확한 문서를 찾으려 하고, 일반적인 질문은 여러 관련 문서를 반환한다. 셋째, 시간 범위는 정보의 시간적 관심을 나타낸다. 최근 정보만 원하는지, 전체 이력이 필요한지를 결정한다. 넷째, 탐색 시작점은 분류 체계에서 어디서부터 탐색을 시작할지를 결정한다. 다섯째, 신뢰도 요구는 정보의 출처나 최신성에 대한 요구 수준이다.

#### 4.2.2 문서 탐색 단계

문서 탐색은 분류 체계를 트리 탐색하듯이 진행된다. 탐색 알고리즘은 먼저 Wiki 루트의 폴더 구조를 읽고 최상위 분류 카테고리를 확인한다. 의도 분석 결과를 바탕으로 가장 관련성이 높아 보이는 카테고리부터 시작한다. 각 폴더에 대해 폴더 이름과 해당 폴더의 문서 제목들을 빠르게 스캔한다. 필요시 폴더의 하위 폴더로 재귀적으로 진입한다. 최종적으로 관련된 문서의 절대 경로를 수집한다.

탐색 효율성을 위해 폴더 인덱스 파일을 활용한다. 각 폴더에 대해 폴더 내 문서 목록과 각 문서의 간단한 요약을 담은 인덱스 파일을 유지한다. 이 인덱스 파일은 문서 변경 시 자동으로 업데이트되어 항상 최신 상태를 반영한다. 인덱스 파일 덕분에 폴더를 열지 않고도 내용 파악이 가능하다.

#### 4.2.3 문서 반환 단계

문서 반환 단계는 탐색 결과를 최종 사용자에게 전달한다. 반환 형식은 세 가지 모드를 지원한다. 경로 반환 모드는 관련 문서의 절대 경로 목록만 반환한다. 요약 반환 모드는 각 문서의 경로와 함께 간단한 요약도 반환한다. 전체 반환 모드는 문서의 전체 내용을 반환한다.

반환 시 중복과 우선순위를 고려한다. 동일하거나 매우 유사한 내용을 가진 문서가 여러 개 있다면 가장 적절한 하나만 반환하거나 함께 반환 여부를 사용자에게 확인한다. 가장 최근 수정된 문서와 원본 정보를 우선적으로 반환한다.

### 4.3 백링크 기반 탐색

OutoWiki의 검색은 단순히 폴더 구조를 따르는 것 이상으로 백링크를 활용한다. 각 문서는 자신이 참조하는 다른 문서와, 자신을 참조하는 문서들의 목록을 명시적으로 포함한다. 이 백링크 구조는 두 가지 방식으로 검색을 지원한다.

첫째, 참조 확장 방식이다. 핵심 문서를 찾은 후 해당 문서가 참조하는 문서들도 함께 로드한다. 예를 들어 사용자 취향 문서를 찾으면, 그 문서에서 참조하는 사용자 프로필 문서도 함께 반환한다. 둘째, 역참조 추적 방식이다. 특정 개념에 대한 문서를 찾을 때, 그 개념을 참조하는 다른 문서들도 찾는다. 이는 정보의 폭을 넓혀 더 완전한 맥락을 제공한다.

## 5. Wiki 운영 규칙

### 5.1 문서命名 및 제목 규칙

OutoWiki의 문서命名은 일관된 규칙을 따른다. 제목은 명확성과 검색 가능성을 위해 구체적으로 작성한다. 예를 들어 음식 취향보다 사용자_Alice_음식_취향_2024가 더 구체적이다. 제목에 날짜를 포함하면 시간적 맥락이 명확해지지만, 날짜가 변경되지 않는 핵심 지식에는 날짜를 포함하지 않는다.

파일 이름은 소문자와 밑줄을 사용한다. 이는 파일 시스템 간의 호환성을 높이고 URL에서 문제가 되는 문자를 피한다. 확장자는 항상 .md로 통일하여 마크다운 형식을 명시한다.

### 5.2 문서 분류 규칙

분류는 폴더 구조를 통해 수행되며, 상위 폴더에서 하위 폴더로 갈수록 구체적인 분류가 된다. 각 분류 폴더에는 해당 분류에 대한 설명을 담은 README.md 파일을 배치하여 분류의 의미를 명확히 한다.

분류 결정 시 중복 분류를 허용하되 명확성을優先한다. 하나의 정보가 여러 카테고리에 속할 수 있다면, 가장 적절한 주 카테고리를 선택하고 다른 카테고리에서는 백링크를 통해 참조한다. 예를 들어 Python 프로그래밍 지식은 knowledge/domain/programming/python.md에도, tools/scripting/python.md에도 관련될 수 있다. 이 경우 주 분류를前者로 하고 후자에서 백링크한다.

### 5.3 문서 병합 및 분할 규칙

OutoWiki는 실제 위키의 운영 방식을 수용하여 체계적인 문서 병합 및 분할을 지원한다. 이 규칙들은 Wikipedia와 나무위키의 검증된 운영 원칙을 기반으로 설계되었다.

#### 5.3.1 분할 규칙 (Splitting)

##### 분할이 필요한 상황

문서 분할은 두 가지 주요 상황에서 고려된다. 첫째, 문서 크기로 인한 분할이다. 문서가 너무 커져 읽기 어렵거나 특정 섹션이 전체 문서에 비해 과도하게 길어질 때 분할을 검토한다. 둘째, 주제 범위로 인한 분할이다. 기존 문서에서 다루는 주제가 범위를 벗어나거나, 하위 주제가 독자적인 주목할 만성을 갖추었을 때 분할을 고려한다.

##### 크기 기반 분할 기준

Wikipedia의 문서 크기 가이드라인을 참고하여 다음과 같은 토큰 기반 기준을 적용한다.

| 토큰 수 (약) | 분할 필요성 |
|-------------|------------|
| 15,000 토큰 초과 | 거의 확실히 분할 필요 |
| 9,000 토큰 초과 | 대체로 분할 필요 (주제 범위로 정당화 가능) |
| 8,000 토큰 초과 | 분할 검토 필요 |
| 6,000 토큰 미만 | 분할 불필요 (단순 길이만으로는 분할 불가) |
| 300 토큰 미만 | 병합 또는 확장 고려 (스텁 문서) |

##### 분할 결정 기준

분할이 적절한지 판단하기 위해 다음 기준을 모두 충족해야 한다. 첫째, 충분한 내용량이다. 분할된 새 문서가 독자적인 백과사전적 내용을 담기에 충분한 분량이어야 한다. 분할 후 스텁(매우 짧은 문서)이 될 예정이라면 분할이 너무 이르다. 둘째, 논리적 독립성이다. 분할 대상 섹션이 메인 문서 없이도 독립적으로 존재할 수 있는 명확한 주제여야 한다. 셋째, 참조 가치이다. 다른 문서에서 자주 참조될 가능성이 있는 내용을 분할하면 백링크 구조를 효과적으로 활용할 수 있다.

##### 분할 절차

분할은 다음 단계를 따라 진행된다. 첫째, 분할 계획 수립이다. 내부 에이전트가 분할 대상 섹션, 새 문서 제목, 메인 문서의 요약 작성 방식을 계획한다. 둘째, 새 문서 생성이다. 분할 대상 내용을 새 문서로 복사하고 적절한 헤더와 개요를 추가한다. 셋째, 메인 문서 업데이트이다. 메인 문서에서 분할된 내용을 제거하고 [[새 문서|링크]] 형식의 요약 섹션으로 대체한다. Wikipedia의 {{Main}} 템플릿과 유사한 역할을 한다. 넷째, 역링크 업데이트이다. 분할된 새 문서에 메인 문서를 역참조로 추가하고, 기존 백링크를 새 문서로 향하도록 업데이트한다. 다섯째, 참조 추적이다. 분할 전 원본 문서를 참조하던 다른 문서들의 링크를 새 문서로 향하도록 수정한다.

##### 분할 후 메인 문서 요약 스타일

Wikipedia의 "Summary style" 규칙을 적용한다. 분할 후 메인 문서는 분할된 주제에 대한 간결한 요약만 포함해야 한다. 요약은 2~3개 섹션 내외가 적당하며, 상세 내용은 하위 문서에 위임한다. 이를 통해 에이전트가 메인 문서만 읽어도 전체 구조를 파악할 수 있다.

#### 5.3.2 병합 규칙 (Merging)

##### 병합이 필요한 상황

문서 병합은 네 가지 주요 상황에서 발생한다. 첫째, 중복 문서 병합이다. 동일하거나 매우 유사한 범위를 다루는 두 개 이상의 문서가 존재할 때 이를 하나의 문서로 통합한다. 둘째, 범위 중첩 병합이다. 관련된 주제를 다루지만 내용이 상당 부분 중첩되는 문서들을 합친다. Wikipedia는 사전을 만들的地方이 아니며, 모든 개념마다 별도의 항목을 만들 필요는 없다. 셋째, 스텁 문서 병합이다. 매우 짧은 문서(2~3문장)가 다른 관련 문서의 범위에 포함될 때 병합한다. 넷째, 문맥 필요 병합이다. 독립적으로 주목할 만성은 갖추었으나 다른 문서와 함께 다루는 것이 독자에게 더 나은 이해를 제공할 때 병합한다.

##### 스텁 문서 정책

Wikipedia의 스텁 정책에 따라, 300토큰 미만의 짧은 문서는 다음 조건 충족 시 즉시 병합할 수 있다. 해당 내용이 다른 관련 문서와 유사하거나 중복된다. 내용이 단순 문구 차이만 다르다. 별도 확장 계획이 없다. 이러한 조건의 문서는 내부 에이전트가 사전 토론 없이 직접 병합할 수 있다.

##### 병합 결정 기준

병합이 적절하지 않은 상황도 존재한다. 첫째, 결과 문서가 과도하게 커지는 경우이다. 병합 결과 문서가 분할 기준을 초과할 경우 병합보다 분할을 고려한다. 둘째, 독립적 발전 가능성이다. 현재 짧더라도 독립적으로 확장될 가능성이 있는 문서는 병합하지 않는다.

##### 병합 절차

병합은 다음 단계를 따라 진행된다. 첫째, 병합 대상 선정이다. 소스 문서(병합될 문서)와 대상 문서(병합받는 문서)를 결정한다. 둘째, 내용 이동이다. 소스 문서의 내용을 대상 문서의 적절한 위치에 통합한다. 내용 편집 시 중복 표현을 제거하고 일관된 톤을 유지한다. 셋째, 리다이렉트 생성이다. 소스 문서를 삭제하고 대상 문서를 향하는 리다이렉트 파일을 생성한다. Wikipedia의 `#REDIRECT [[대상문서]]`와 유사하다. 넷째, 백링크 업데이트이다. 소스 문서를 참조하던 다른 문서들의 링크를 대상 문서로 업데이트한다. 다섯째, 역링크 동기화이다. 대상 문서의 역링크 목록에 병합된 내용이 반영되었음을 기록한다.

##### 병합 우선순위 규칙

병합 시 충돌하는 정보의 처리 규칙은 다음과 같다. 최신 정보 우선이다. 동일 주제에 대해 다른 내용이 있을 때 더 최근의 정보를 우선 적용한다. 신뢰도 기준이다. 신뢰도 높은 출처의 정보가 신뢰도 낮은 출처의 정보를 대체한다. 출처가 없는 내용은 출처가 있는 내용으로 대체한다. 충돌 정보 처리이다. 직접 대체할 수 없는 충돌 정보는 별도의 "논의" 섹션이나 관련 하위 문서로 분리하여 원본 문서를 참조 목록에 포함시킨다.

#### 5.3.3 분리된 문서의 상호 참조 구조

실제 위키에서처럼, OutoWiki는 분할된 문서들 사이의 명시적 참조 구조를 관리한다. 이 구조는 세 가지 구성요소로 이루어진다.

##### 하위 문서에서 상위 문서로

분할된 각 하위 문서는 자신의 메인 문서를 명확히 참조해야 한다. 문서 헤더의 related 필드나 상단에 "본 문서는 [[상위 문서]]의 일부입니다"와 같은 설명을 포함한다. 이는 문서의 맥락을 파악하고 역방향 탐색을 가능하게 한다.

##### 상위 문서에서 하위 문서로

상위 문서는 분할된 각 하위 문서를 명시적으로 참조한다. 분할된 주제를 다루는 섹션에서 [[하위 문서]] 링크를 제공하고, 필요하다면 {{Main|하위 문서}} 형식의 템플릿을 사용한다. Wikipedia에서와 같이, 메인 문서의 해당 섹션은 하위 문서의 내용을 요약한 형태로 제공된다.

##### 분할 패턴의 실제 사례

다음은 일반적인 분할 패턴의 예시이다.

애니메이션 문서의 경우, 메인 문서는 줄거리, 제작 배경, 방영 정보를 담고, 별도의 등장인물 문서가 등장인물 목록과 상세 정보를 담는다. 둘 사이의 참조는 다음과 같다. 메인 문서: "자세한 내용은 [[문서명/등장인물]]을 참조하세요." / 등장인물 문서: "이 문서는 [[문서명]]의 일부입니다."

웹사이트 문서의 경우, 메인 문서는 사이트 개요, 역사, 주요 기능을 담고, 별도의 문서가 문제점 및 논란, 차단风波, 법적 분쟁 등을 다룬다. 둘 사이의 참조는 다음과 같다. 메인 문서: "논란의 자세한 내용은 [[문서명/논란]]을 참고하세요." / 논란 문서: "이 문서는 [[문서명]]의 일부이며, [[문서명/역사]]의 시기와 관련됩니다."

도구 문서의 경우, 메인 문서는 도구 개요와 기본 사용법을 담고, 별도의 문서가 고급 사용법, 제한사항, 버전 변천史 등을 다룬다. 에이전트가 도구의 전체 구조를 빠르게 파악하고 필요시 상세 문서로 이동할 수 있다.

#### 5.3.4 분할/병합 후 처리 규칙

나무위키의 문서 관리 정책을 참고하여 다음과 같은 사후 처리 규칙을 적용한다.

##### 깨진 링크 정상화

분할이나 병합 완료 후 3시간 이내에 모든 깨진 링크를 수정해야 한다. 기존 문서명을 참조하던 모든 백링크를 새 위치로 업데이트한다. 리다이렉트를 생성한 경우, 해당 리다이렉트가 올바른 대상을 가리키는지 확인한다.

##### 참조 무결성 보장

참조 대상이 삭제되거나 이동된 경우, 해당 참조를 제거하거나 대체 참조를 제공한다. 가능한 경우, 삭제된 내용과 유사한 다른 문서를 대체 참조로 지정한다. 참조 변경 시 관련된 모든 문서의 역링크 목록을 동기화한다.

##### 히스토리 관리

분할이나 병합 시 원본 문서의 변경 이력을 보존한다. WikiStore는 내부적으로 파일 시스템의 버전을 관리하므로, 원본 문서는 히스토리에서 확인 가능해야 한다. 병합 시에는 모든 원본 문서의 기여를 병합 결과에 반영한다.

### 5.4 문서 참조 규칙

Wiki 내 문서 간 참조는 표준 마크다운 링크 형식을 사용한다. 같은 Wiki 내 문서를 참조할 때는 상대 경로를, 외부 리소스를 참조할 때는 절대 URL을 사용한다. 백링크 섹션은 자동으로 생성되어 문서 하단에 위치하며, 해당 문서를 참조하는 모든 문서의 목록을 포함한다.

참조 무결성을 위해 참조 변경 시 관련 문서의 참조 목록도 업데이트된다. 참조 대상이 삭제되면 해당 참조는 제거되고, 가능하다면 대체 참조가 제공된다.

## 6. 기술 구현 설계

### 6.1 프로젝트 구조

OutoWiki 프로젝트는 다음과 같은 디렉토리 구조를 갖는다. 최상위에 라이브러리 핵심 코드인 outowiki/ 폴더가 위치하며, tests/ 폴더에는 단위 테스트와 통합 테스트가 포함된다. docs/ 폴더에는 사용자 가이드와 개발자 문서가 있으며, examples/ 폴더에는 사용 예시가 포함된다. pyproject.toml은 프로젝트 구성과 의존성을 정의한다.

outowiki/ 폴더 내부의 구조는 다음과 같다. core/ 폴더에는 설정 관리, Wiki 저장소 인터페이스, 기본 유틸리티가 포함된다. modules/ 폴더에는 기록 모듈, 검색 모듈, 내부 에이전트가 포함된다. models/ 폴더에는 데이터 모델 클래스와 스키마 정의가 포함된다. prompts/ 폴더에는 에이전트가 사용하는 프롬프트 템플릿이 포함된다. utils/ 폴더에는 마크다운 파싱, 파일 시스템 유틸리티 등의 헬퍼 함수가 포함된다.

### 6.2 핵심 클래스 설계

#### 6.2.1 OutoWiki 클래스

OutoWiki는 라이브러리의 메인 진입점이자 facade 클래스이다. 이 클래스는 모든 기능을 조율하고 일관된 인터페이스를 제공한다.

```python
class OutoWiki:
    def __init__(self, config: WikiConfig) -> None:
        """OutoWiki 인스턴스를 초기화한다."""

    def configure(self, provider: LLMProvider, wiki_path: str) -> None:
        """LLM 공급자와 Wiki 경로를 설정한다."""

    def record(self, content: RawContent, metadata: Optional[RecordMetadata] = None) -> RecordResult:
        """정보를 Wiki에 기록한다."""

    def search(self, query: SearchQuery) -> SearchResult:
        """Wiki에서 정보를 검색한다."""

    def get_document(self, path: str) -> WikiDocument:
        """지정된 경로의 문서를 반환한다."""

    def update_document(self, path: str, content: str) -> None:
        """문서를 업데이트한다."""

    def delete_document(self, path: str) -> None:
        """문서를 삭제한다."""

    def list_categories(self) -> List[Category]:
        """최상위 분류 목록을 반환한다."""
```

#### 6.2.2 Recorder 클래스

Recorder 클래스는 정보 기록 기능을 담당한다. 내부적으로 세 단계를 순차적으로 실행한다.

```python
class Recorder:
    def __init__(self, wiki: WikiStore, agent: InternalAgent) -> None:
        """Recorder를 초기화한다."""

    def record(self, content: RawContent, metadata: Optional[RecordMetadata] = None) -> RecordResult:
        """정보를 기록하고 결과를 반환한다."""

    def _analyze(self, content: RawContent) -> AnalysisResult:
        """입력 정보를 분석한다."""

    def _plan(self, analysis: AnalysisResult, wiki_state: WikiState) -> List[Plan]:
        """분석 결과를 바탕으로 계획을 수립한다."""

    def _execute(self, plans: List[Plan]) -> ExecutionResult:
        """계획을 실행한다."""
```

#### 6.2.3 Searcher 클래스

Searcher 클래스는 정보 검색 기능을 담당한다. 탐색 알고리즘과 인덱싱 로직을 포함한다.

```python
class Searcher:
    def __init__(self, wiki: WikiStore, agent: InternalAgent) -> None:
        """Searcher를 초기화한다."""

    def search(self, query: SearchQuery) -> SearchResult:
        """검색 쿼리를 처리하고 결과를 반환한다."""

    def _analyze_intent(self, query: SearchQuery) -> IntentAnalysis:
        """검색 의도를 분석한다."""

    def _explore(self, intent: IntentAnalysis) -> List[str]:
        """분류 체계를 탐색하여 관련 문서를 찾는다."""

    def _return_results(self, paths: List[str], mode: ReturnMode) -> SearchResult:
        """탐색 결과를 반환 형식으로 가공한다."""
```

#### 6.2.4 WikiStore 클래스

WikiStore 클래스는 Wiki 저장소에 대한 저수준 인터페이스를 제공한다. 파일 시스템 조작과 문서 관리의 기본 연산을 캡슐화한다.

```python
class WikiStore:
    def __init__(self, root_path: str) -> None:
        """Wiki 저장소를 초기화한다."""

    def read_document(self, path: str) -> WikiDocument:
        """문서를 읽는다."""

    def write_document(self, path: str, content: WikiDocument) -> None:
        """문서를 쓴다."""

    def delete_document(self, path: str) -> None:
        """문서를 삭제한다."""

    def list_folder(self, folder_path: str) -> FolderContent:
        """폴더 내용을 나열한다."""

    def create_folder(self, folder_path: str) -> None:
        """폴더를 생성한다."""

    def delete_folder(self, folder_path: str) -> None:
        """폴더를 삭제한다."""

    def get_index(self, folder_path: str) -> FolderIndex:
        """폴더 인덱스를 반환한다."""

    def update_backlinks(self, path: str, backlinks: List[str]) -> None:
        """문서의 백링크를 업데이트한다."""
```

### 6.3 데이터 모델 정의

OutoWiki는 명확한 데이터 모델을 사용한다. RawContent는 기록 모듈에 입력되는 원시 콘텐츠를 표현한다. WikiDocument는 Wiki에 저장되는 문서를 표현한다. SearchQuery는 검색 모듈에 입력되는 검색 쿼리를 표현한다. SearchResult는 검색 결과를 표현한다. Plan은 수정 계획의 기본 클래스로, 생성, 수정, 병합, 분할, 삭제 계획이 이를 상속한다.

각 데이터 모델은 타입 힌트와 기본값을 제공하여 코드 작성의 편의성을 높인다. 또한 Pydantic 모델을 사용하여 런타임 시 데이터 검증이 이루어진다.

### 6.4 LLM 공급자 추상화

OutoWiki는 다양한 LLM 공급자를 지원하기 위해 Provider 추상 클래스를 정의한다. 구체적인 구현으로는 OpenAIProvider와 AnthropicProvider가 제공되며, 사용자는 필요에 따라 커스텀 공급자도 구현할 수 있다.

```python
class LLMProvider(ABC):
    @abstractmethod
    def complete(self, prompt: str, **kwargs) -> str:
        """프롬프트에 대한 완성 결과를 반환한다."""
        pass

    @abstractmethod
    def complete_with_schema(self, prompt: str, schema: dict, **kwargs) -> dict:
        """스키마에 맞는 구조화된 결과를 반환한다."""
        pass

class OpenAIProvider(LLMProvider):
    def __init__(self, api_key: str, base_url: str, model: str, max_tokens: int):
        """OpenAI 호환 API 공급자를 초기화한다."""

class AnthropicProvider(LLMProvider):
    def __init__(self, api_key: str, base_url: str, model: str, max_tokens: int):
        """Anthropic API 공급자를 초기화한다."""
```

## 7. 사용 예시 및 시나리오

### 7.1 기본 사용 패턴

OutoWiki의 기본 사용은 매우 간단하다. 먼저 라이브러리를 임포트하고 설정을 초기화한다. 그런 다음 record() 메서드를 사용하여 정보를 기록하고, search() 메서드를 사용하여 정보를 검색한다.

```python
from outowiki import OutoWiki, WikiConfig

# 설정
config = WikiConfig(
    provider="openai",
    base_url="https://api.openai.com/v1",
    api_key="your-api-key",
    model="gpt-4",
    max_tokens=4000,
    wiki_path="./my_wiki"
)

# Wiki 초기화
wiki = OutoWiki(config)

# 정보 기록
result = wiki.record("""
User: 나는 파이썬으로 웹 개발을 하고 싶어
Agent: 파이썬으로 웹 개발을 하려면 Flask나 Django 같은 프레임워크를 사용할 수 있습니다.
""")

# 정보 검색
results = wiki.search("사용자가 원하는 웹 개발 방법")
print(results.paths)
```

### 7.2 에이전트 메모리 통합 시나리오

OutoWiki의 진정한 힘은 AI 에이전트에 통합될 때 발휘된다. 에이전트의 루프 내에서 OutoWiki를 활용하는 예시를 살펴보자.

에이전트가 사용자와 대화하며 작업을 수행한다고 가정하자. 각 대화 턴 후 에이전트는 OutoWiki에 주요 정보를 기록한다. 사용자 취향, 선호하는 작업 방식, 성공하거나 실패한 접근법 등이 기록된다. 이후 사용자가 새로운 작업을 요청하면, 에이전트는 먼저 Wiki를 검색하여 관련 정보를 확인한다. 이를 통해 에이전트는 사용자의 과거 선호도를 기억하고 더욱 개인화된 서비스를 제공할 수 있다.

```python
class AgentWithMemory:
    def __init__(self):
        self.wiki = OutoWiki(config)

    def process_turn(self, user_input: str, agent_response: str):
        # 대화 기록
        self.wiki.record({
            "type": "conversation",
            "content": f"User: {user_input}\nAgent: {agent_response}",
            "context": self.current_task
        })

        # 사용자 피드백 감지
        if "고마워" in user_input or "잘했어" in user_input:
            self.wiki.record({
                "type": "learning",
                "content": f"Successful approach: {agent_response[:100]}...",
                "context": self.current_task
            })

    def before_task(self, task: str):
        # 과거 관련 정보 검색
        results = self.wiki.search(task)
        if results.paths:
            context = self.wiki.get_documents(results.paths)
            self.context = context
```

### 7.3 지식 베이스 구축 시나리오

OutoWiki를 사용하여 에이전트의 지식 베이스를 구축하는 예시도 가능하다. 웹 검색 결과, 문서 읽기 결과, API 응답 등을 Wiki에 기록할 수 있다.

```python
# 웹 검색 결과 기록
search_results = web_search("Python best practices 2024")
wiki.record({
    "type": "external_knowledge",
    "content": str(search_results),
    "source": "web_search",
    "topic": "python_best_practices"
})

# 문서 읽기 결과 기록
doc_content = read_pdf("python_guide.pdf")
wiki.record({
    "type": "document",
    "content": doc_content.summary,
    "source": "python_guide.pdf",
    "topic": "python_programming"
})
```

## 8. 개발 로드맵

### 8.1 1단계: 핵심 기능 구현

첫 번째 단계에서는 OutoWiki의 핵심 기능을 구현한다. Wiki 저장소 클래스, 기본 설정 시스템, 폴더 기반 분류 관리 기능을 먼저 구현한다. 이어서 기본 검색 기능—폴더 네비게이션, 문서 경로 반환—을 구현한다. 마지막으로 기본 기록 기능—문서 생성, 단순 수정—을 구현한다. 이 단계의 목표는 기본적인 Wiki 관리 기능이 동작하는 것이다.

### 8.2 2단계: 내부 에이전트 통합

두 번째 단계에서는 내부 에이전트를 통합한다. 분석 에이전트—입력 정보 유형 분류, 핵심 정보 추출—를 구현한다. 계획 수립 에이전트—문서 생성/수정/병합/분할 계획—를 구현한다. 검색 에이전트—의도 분석, 탐색 전략 수립—를 구현한다. 백링크 관리 시스템도 함께 구현하여 문서 참조의 무결성을 보장한다.

### 8.3 3단계: 고급 기능 추가

세 번째 단계에서는 고급 기능을 추가한다. 문서 병합 및 분할 자동화—문서 크기 모니터링, 자동 분할 제안—를 구현한다. 인덱스 시스템—폴더 인덱스 파일, 빠른 탐색 지원—을 구현한다. 변경 이력 추적—버전 관리, 롤백 기능—을 구현한다. 다중 공급자 지원—Anthropic 등 추가 공급자—도 이 단계에서 구현한다.

### 8.4 4단계: 최적화 및 안정화

마지막 단계에서는 최적화와 안정화에 집중한다. 성능 profiling과 최적화를 수행한다. 오류 처리와 예외 상황 관리，完善한单元测试와 통합测试를 작성한다. 문서화를 완성하여 사용자 가이드와 API 참조를 작성한다.

## 9. 기대 효과 및 활용 분야

### 9.1 에이전트 메모리 관리의 혁신

OutoWiki는 AI 에이전트의 메모리 관리 방식을 혁신한다. 기존 방식이 벡터 임베딩에 의존했다면, OutoWiki는 구조화된 Wiki 체계를 통해 예측 가능하고 해석 가능한 메모리 관리를 제공한다. Hermes Agent의 자체 학습 기능을 일반화하고, MemGPT의 계층적 메모리 개념을Wiki 구조로 구현한다.

### 9.2 활용 분야

OutoWiki는 다양한 분야에서 활용될 수 있다. 개인 AI 어시스턴트는 사용자의 선호도와 대화 이력을Wiki에 저장하여 장기적으로 개인화된 서비스를 제공한다. 팀 협업 에이전트는 팀원의 역할, 선호도, 진행 중인 프로젝트 정보를Wiki에 공유하여 팀원 간 협업을 원활하게 한다. 자율주행 에이전트는 작업 히스토리, 성공/실패 패턴, 효과적인 전략을Wiki에 축적하여 지속적으로 스스로를 개선한다. 고객 서비스 에이전트는 고객별 상호작용 이력과 선호도를Wiki에 기록하여 각 고객에 맞는 맞춤형 서비스를 제공한다.

### 9.3 경쟁력 분석

OutoWiki의 경쟁력은 명확하다. RAG 시스템 대비 인프라 단순성, 자체 학습 기능, Wiki 규칙 기반의 체계적 관리라는 장점이 있다. 기존 위키 도구 대비 AI 통합, 자동 분류, 자체 문서화 기능이라는 차별점이 있다. 단순 KV 저장소 대비 구조화된 분류, 검색 가능성, 인간과 AI 모두에게 읽기 쉬운 형식이라는advantages이 있다.

## 10. 결론

OutoWiki는 AI 에이전트를 위한 범용 Wiki 라이브러리로서, 에이전트의 모든 지식—기억, 학습, 역사, 도구 사용법—을 체계적으로 관리한다. 실제 위키의 운영 원칙을 수용하여, 문서의 분류, 병합, 분할, 참조, 수정을 자동화한다. 폴더 기반 직관적 분류와 마크다운 형식을 통해 인간과 AI 모두에게 가독성을 보장한다.

내부 에이전트에 의한 자율적 정보 처리—분석, 계획, 반영—를 통해Wiki는 스스로 진화하고 성장한다. 설정 선언, 정보 기록, 정보 검색이라는 세 가지 단순한 인터페이스를 통해 사용자는 복잡한 내부 구현을 알 필요 없이 강력한 기능을 활용할 수 있다.

OutoWiki는 단순한 저장소를 넘어 에이전트의 두 번째 뇌, 지속적 학습의 기반, 협업의 연결고리가 될 것이다. 에이전트의 모든 것을 기록하는 Wiki—그것이 OutoWiki의 비전이다.
